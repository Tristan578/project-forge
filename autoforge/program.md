# AutoForge Program — Compound AI Action Optimization

> This file is the autoresearch "program.md" — it tells Claude Code what to
> experiment with, what constraints to respect, and how to evaluate results.
> Humans edit this file to steer the loop. The agent reads it at the start
> of every experiment cycle.

## Goal

Improve the quality of scenes generated by SpawnForge's compound AI actions
(`create_scene_from_description`, `create_level_layout`, `setup_character`,
`configure_game_mechanics`, `arrange_entities`, `apply_style`).

Quality is measured by a composite score: heuristic analysis (0-60) plus
vision evaluation (0-50) = 0-110 total, averaged across 10 benchmark
prompts representing diverse game scene types.

## Editable Surface

You may ONLY modify these files:

- `web/src/lib/chat/handlers/compoundHandlers.ts` — primary target
- `web/src/lib/chat/handlers/sceneManagementHandlers.ts` — scene creation helpers
- `web/src/lib/chat/handlers/gameplayHandlers.ts` — gameplay setup helpers
- `web/src/lib/chat/context.ts` — scene context passed to AI

Do NOT modify: the engine (Rust/WASM), React UI components, Zustand store
slices, test files, MCP manifest, or any other handler files.

## Experiment Protocol

Each experiment follows this exact sequence:

1. Read this file and `autoforge/results/loop-state.json` for context
2. Form ONE hypothesis about what to change and why
3. Make ONE focused change in the editable surface files
4. Run: `cd autoforge && npx tsx scripts/run-eval.ts`
5. If heuristic score >= baseline: run with `--vision` flag
6. If total score > best score: KEEP (commit). Otherwise: DISCARD (git checkout)
7. Log the result and repeat

**One concept per experiment.** Don't combine multiple changes — it makes
attribution impossible when analyzing what worked.

## What To Experiment With

### Material Selection (high impact)
- Which material presets are assigned to common object types?
- Does matching material to context improve scores? (e.g., "weathered stone" for
  dungeons vs "clean stone" for modern scenes)
- Do emissive materials on light-emitting objects (torches, screens) improve
  lighting scores?

### Lighting Setup (high impact)
- Default light count, type, color temperature, intensity per scene mood
- Warm point lights near fire/torches vs cool for sci-fi
- Ambient light level — too high kills mood, too low kills readability
- Does adding fog matching the scene mood improve vision scores?

### Entity Arrangement (medium impact)
- Spacing between objects — cluttered vs sparse
- Grouping related objects (parent-child hierarchy)
- Focal point placement — is the "star" entity placed where the camera looks?
- Edge padding — objects at scene boundary vs centered compositions

### Camera Positioning (medium impact)
- Default camera yaw/pitch/radius after scene creation
- Does framing the scene nicely improve vision scores?
- Different defaults for indoor vs outdoor scenes

### Post-Processing (medium impact)
- Bloom intensity per mood (fantasy=high, industrial=low)
- Color grading per environment type
- Fog distance and density matching environment

### Audio Defaults (low-impact on visual score, high on completeness)
- Adding ambient audio sources on relevant entities
- Spatial audio placement on sound-producing objects

### Physics Defaults (low-impact on visual score, high on completeness)
- Static bodies on environment objects
- Appropriate mass/friction for interactive objects
- Colliders on gameplay-relevant entities

## Constraints

- Every scene must boot without errors: `npx tsc --noEmit` and `npx eslint --max-warnings 0` must pass
- Every scene must have at least 1 non-ambient light source
- Materials must come from existing presets OR valid StandardMaterial params
- Do not add new MCP commands — only tune orchestration of existing ones
- Do not import new npm packages into the web/ directory
- Keep changes small and single-concept for clear attribution
- Preserve the function signatures and exports of all handler functions
- The `mulberry32` seeded RNG must remain deterministic for reproducibility
